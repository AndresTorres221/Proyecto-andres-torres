## ¿Qué hice?

Se creó el repositorio del proyecto en GitHub y se configuró la estructura inicial de carpetas y archivos utilizando VS Code. Posteriormente, se desarrolló y documentó la fase de análisis del proyecto, incluyendo el planteamiento del problema, espina de pescado, alcance, objetivos, técnicas de elicitación y casos de uso.

También se agregaron y organizaron archivos de apoyo como `README.md`, `BITACORA.md` y `DECISIONES.md` para mantener la documentación y trazabilidad del proyecto.

Adicionalmente, se inició la fase de diseño del sistema, específicamente la Fase 1: Arquitectura, donde se estructuraron los primeros documentos y diagramas, y se comenzó la construcción de la tabla de contexto del sistema.

---

## ¿Qué problema encontré?

Durante la organización inicial del repositorio se presentaron inconvenientes con la visualización de carpetas vacías en GitHub, lo que generó confusión en la estructura del proyecto. Además, algunos cambios realizados en ciertos archivos no se reflejaban correctamente debido a que no habían sido guardados antes de ejecutar los comandos de Git.

También se presentó un conflicto de versiones al realizar un `git pull`, ya que existían cambios diferentes entre el repositorio local y el remoto, generando un conflicto en el archivo de la tabla de contexto.

---

## ¿Cómo lo resolví?

Se utilizaron archivos `.gitkeep` para permitir que GitHub mostrara las carpetas vacías y se verificó constantemente el guardado correcto de los archivos antes de realizar commits y pushes.

El conflicto de Git se resolvió manualmente editando el archivo afectado, eliminando las marcas de conflicto (`<<<<<<<`, `=======`, `>>>>>>>`) y conservando la versión correcta del contenido. Posteriormente, se realizó el `git add`, `git commit` y `git push` para sincronizar el repositorio.

Finalmente, se reorganizó la estructura del repositorio desde VS Code y se revisó la ubicación de cada documento para asegurar un mejor orden y trazabilidad del proyecto.
